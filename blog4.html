Project Objective

The main goal of this system is to simplify the course registration process for students while allowing advisors and administrators to manage course offerings, class schedules, and room allocations. It focuses on solving the problem of overlapping class times and improving scheduling accuracy.

Conceptual Modeling

We chose a centralized approach where all types of users (students, advisors) are integrated into one system. This design ensures consistency, easy maintenance, and avoids data duplication.

Key design principles include:

  • Overlapping User Needs: Students and advisors both need access to course data and timetables.
  • Manageable Complexity: The system includes important entities like Student, Course, Timetable, but keeps relationships straightforward.
  • Consistency & Integration: All schedules, room data, and registrations are stored in one unified database.
  • Simplified Maintenance: Any updates to course timings, sections, or classrooms can be made from a single location.

Entities and Relationships

The system is built on the following main entities:

  • Advisor: Stores ID, Name, and Email of academic advisors.
  • Student: Holds enrollment number, name, and email.
  • Course: Contains Course ID, Name, Credit Hours, and advisor information.
  • Course Offering: Semester and Year when a course is offered.
  • Timetable: Stores class timings, days, sections, and assigned classrooms.
  • Classroom: Holds room number, building, and seating capacity.
  • Register: Keeps record of student-course-section registrations.

Normalization Process

To ensure data integrity and eliminate redundancy, we applied all three normal forms:

First Normal Form (1NF)

All attributes were made atomic, and repeating groups were removed. Each table had a primary key.

Second Normal Form (2NF)

We removed partial dependencies. For example, in the Register table, each non-key column depends on the full composite key.

Third Normal Form (3NF)

We eliminated transitive dependencies. For instance, advisor and student personal information was split into separate tables to ensure every non-key attribute depends only on the primary key.

Final Table Structure

After normalization, the database schema included the following:

  • Advisor(A_ID, A_Email)
  • AdvisorInformation(A_Email, A_Name)
  • Student(S_Enrollment, S_Email)
  • StudentInformation(S_Email, S_Name)
  • Course(C_ID, Course_Name, A_ID, Semester, Year)
  • CourseInformation(Course_Name, Credit_Hour)
  • TimeTable(Day, Section_Name, Start_Time, End_Time, C_ID, RoomNo)
  • ClassRoom(RoomNo, Capacity)
  • BuildingInfo(RoomNo, Building)
  • Register(S_Enrollment, C_ID, Section_Name)

SQL Table Creation (DDL)

We implemented the database using SQL DDL scripts. Here are examples of key table creation statements:

CREATE TABLE Advisor (
  A_ID INT PRIMARY KEY,
  A_Email VARCHAR(30)
);

CREATE TABLE AdvisorInformation (
  A_Email VARCHAR(30) PRIMARY KEY,
  A_Name VARCHAR(30)
);

CREATE TABLE Student (
  S_Enrollment INT PRIMARY KEY,
  S_Email VARCHAR(30)
);

CREATE TABLE Register (
  S_Enrollment INT,
  C_ID VARCHAR(10),
  Section_Name VARCHAR(5),
  PRIMARY KEY (S_Enrollment, C_ID, Section_Name),
  FOREIGN KEY (S_Enrollment) REFERENCES Student(S_Enrollment)
);

These scripts were executed using MySQL Workbench and follow best practices for relational integrity.
